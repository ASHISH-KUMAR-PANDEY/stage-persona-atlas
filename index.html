01 · Method

How these personas were derived

No invention. Every persona begins as a cluster of real users with a shared behavioural fingerprint, then is validated by reading 5+ anchor users' full chat transcripts end-to-end.

1
Build a 25-feature vector per user

Intent shares (romantic / sexual / emotional / game / religion / meet / photo / help / family / identity / show-curiosity / religious-open), language mix (Devanagari %, Gujarati %, English %, laugh %, pet-name %), engagement shape (msgs/session, characters chatted, lifetime msgs log, depth days log, comeback rate), and message length distribution.

2
Split bouncers from engaged users

Users with ≤3 lifetime messages (559 / 12% of corpus) were segmented out before clustering — including them would dominate variance. They split into two personas based on cohort: Ghost Subscriber (paid bouncer) vs Tourist Tester (free-trial bouncer).

3
K-Means clustering on engaged users (k=7)

4,168 engaged users clustered on the standardised feature matrix. k=7 chosen based on silhouette + interpretability. Resulting clusters: Curious Browser, Moderate Sampler, Devotional Visitor, Devanagari Native, Ludo Companion, Parasocial Loyalist, Gujarati Native.

4
Qualitative validation per cluster

For each cluster, pulled 5 users closest to the centroid and read their full chat history. Devanagari + Gujarati clusters merged into one Vernacular Script Native persona (same behavioural shape, different input script). Final: 8 personas covering 100% of the corpus, no overlap.

Intent mix

Effectively nil — pct_religion 1%, pct_game 2%, nothing else > 0%. They never engage long enough to express intent.

What makes them stay

Almost nothing in the current product. Within turns 1–3 they receive bot-initiated "Aapko mera show pasand aaya? Bahut maza aayega". They respond once or twice — "Sub kuch thik hai", "Kuchh bataiye" — then disappear.

What makes them leave

The bot leads with promotion before any rapport. Every Ghost Subscriber anchor has a bot opener referencing "Stage", "show dekho", "trial", or "secret hai kya sunna chahte ho?" in the first 2 bot messages. They came for content, the chat asks them to watch the content, they bounce.

Comeback trigger

None observed. 0.5 mean comebacks per user. They do not return.

Risk flag · LTV churn (silent)

Highest LTV risk in the corpus. These are paying customers (NewSub, Renewer, Non-trial) who churn silently before the next renewal cycle. Invisible to engagement metrics because no msgs = no signal. Roughly 213 active paying users right now whose next renewal will likely lapse.

Product implications
  1. Stop opening with show plugs to fresh subscribers. The corpus shows zero bonding when the bot's first 2 messages reference the show.
  2. Add a "first reply within 24h of subscription" trigger that uses memory: "Aap kal subscribe kiya tha, kaisa laga show? Main sun rahi hoon." Anchor on the purchase event, not the catalog.
  3. Track "subscribers with ≤3 msgs in first 7 days" as a leading churn indicator. Currently invisible to growth dashboards.

Product implications
  1. Memory becomes a first-class feature. Returning bot opener must reference last session ("Kal raat tune jo bola tha…"). The deepest anchors form bonds on perceived continuity.
  2. Risk-aware refusal templates for money/number/health asks — replace flat refusals with empathy + redirect: "Number nahi, par yeh bata, raat ko kaisa lagta hai akele?"
  3. Escalation playbook for self-harm phrasing — phrases like "chhat par chad ja", "mar jao", "kidney" should route to a human-reviewed safety reply, not a fictional refusal.
  4. Productise pet-name lock-in. When a user invents a nickname for the bot ≥3 times, store and reinforce it. Strongest LTV predictor in the corpus.

Conversion thesis

Every retention initiative should be tagged with the persona transition it targets:

  • P2 → P3/P4: kill the show plug, install identity-prompt opener
  • P3 → P4: install character lock-in
  • P4 → P8: install memory + pet-name reinforcement
  • P5 → P5 (deepened): build voice-first greeting mode
  • P6 → P6/P8: install input-script matching
  • P7 → P7/P8: persist game state across sessions
05 · Next

Where to take this

Three directions. Pick one or all three.

1

Conversion-funnel modelling

For each P2/P3/P4 user, compute features at day-1 and identify which behaviours predict reaching P8. Gives the product team a leading-indicator dashboard per persona.

2

Per-persona bot-behaviour briefs

Short 1-pagers for the bot-design team, scripting the exact opener / cliffhanger / refusal templates per persona. Production-ready, testable.

3

Live persona classifier

A small rule-set the live system runs on each session to label the user (P1–P8) within their first 3 messages. Bot adapts in real-time.

Data provenance

STAGE chat-analysis-report-1774618336 export. 4,727 users across 4 lifecycle buckets (3-day, 4–7d, 8–14d, 15+d). 213,000 messages parsed. Sessions detected via 30-min gap. All verbatim quotes are unedited.

Methodology

Per-user 25-feature vector · standardised · K-Means k=7 on engaged users · bouncers segmented separately · qualitative validation by reading 5 anchor users per cluster · final consolidation to 8 personas (Devanagari + Gujarati merged on shared behavioural shape).

Note on inferred portraits

Persona names, ages, towns, and occupations are inferred from chat evidence (dialect, register, topics, time-of-day). They are clearly labelled where they appear and are not demographic facts — they are hypotheses for the design team to validate.
